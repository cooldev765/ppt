Context
What is PowerDonor?
The problem we're solving and why this platform exists.
The problem
Hundreds of billions of dollars sit in Donor Advised Funds — money already committed to charity but never deployed. Donors don't know where to give. The process feels overwhelming.
The solution
PowerDonor interviews donors to understand their values and interests, then matches them to charities from a database of 1.9M nonprofits. Dafney is our AI interviewer.
The frontend is the donor's entire experience — from the welcome screen through the interview to seeing their matched charities.

Interview flow
Responsibility split — backend and frontend
How state and logic are divided between the two layers.

Backend owns

Flow order — what question comes next
Loop logic — which categories still need Q4a / Q4b
Skip rules — Q4b groups hidden for Environment
Validation — Q3 must be answered before Q4a
Question content — reads from question_bank.json
Answer storage — saves to donors.profile_data

Frontend owns

Rendering the question (layout, animation, UI components)
Input handling (checkboxes, text fields, sliders)
TTS playback — calls /api/tts with tts_prompt received
Progress indicator ("Step 3 of 11")
Loading states between questions
localStorage for session_id persistence
Any changes to flow logic, validation rules, or question content should go through the backend — happy to discuss and iterate on any of these together.
Question types
Six input types — each needs its own UI component
The input_type field in each question object tells you what to render.
free_text
Open textarea, optional. Used for Q1 initial preferences.
→ Textarea + skip button
multi_select
Choose up to N options. Some have a free-text "other" option.
→ Checkbox grid + optional text input
multi_select_with_text
Some options trigger a text field (e.g. "City" → "Which city?").
→ Checkboxes + conditional text inputs
multi_select_grouped
Options organized into collapsible groups (age, race, etc.).
→ Grouped checkboxes with headers
single_select_with_numeric
One option triggers a numeric input (exact dollar amount).
→ Radio buttons + conditional number input
pps_generation
Special: shows generated PPS text, donor edits, then saves.
→ Editable textarea + save button
Read question_bank.json before building any component — it shows exact field names, option shapes, and skip labels for every question.
Interview flow — detail
Q4a, Q4b, Q4c repeat for each cause selected
If the donor picks 3 cause categories in Q3, the next block runs 3 times — once per cause.
Example — donor selects Education, Health, Environment
Q3Donor selects Education, Health, Environment
Q4a×3Geography for Education → Geography for Health → Geography for Environment
Q4b×3Population for Education → Population for Health → (filtered) for Environment
Q4c×3Subcategory for Education → for Health → for Environment
Q5→Q10Engagement, budget, PPS, results count, next steps
For Environment, Q4b filters out age, family, occupation, and identity groups — only community type (Rural/Tribal/Urban) shows. The backend sends the filtered options — the frontend renders what it receives.

Session handling
How sessions work
Simple conventions to keep client and backend in sync.
session_idA UUID returned by /api/interview/start. Store it in localStorage immediately. It identifies the donor for the entire session.
Every requestPass session_id in the request body — not as a header, not as a cookie. Every POST to /api/interview/answer, /api/pps/*, and /api/matching/* must include it.
On reloadRead session_id from localStorage. If present, call GET /api/interview/session/{id} to restore state. If absent, start fresh.
No accountsMVP has no login, no auth, no passwords. The session_id is the only identity. Clear it to start a new interview.
UUID pattern validation is enforced on the backend. Always use the UUID from /api/interview/start.
